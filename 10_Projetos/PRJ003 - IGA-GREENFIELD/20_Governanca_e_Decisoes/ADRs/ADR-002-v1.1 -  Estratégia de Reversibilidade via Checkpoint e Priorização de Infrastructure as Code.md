## Sumário Executivo

Este ADR estabelece duas decisões arquiteturais fundamentais para o PRJ003:

1. **Reversibilidade obrigatória via Checkpoint Hyper-V**: Toda GMUD técnica deve criar um Ponto de Verificação antes da execução, sendo este o mecanismo oficial de rollback em caso de falha.
2. **Priorização de Infrastructure as Code (IaC)**: Toda implementação deve priorizar a criação e execução de scripts automatizados em detrimento de comandos manuais, garantindo documentação, reuso e reprodutibilidade.

---

## Contexto

### Problema Observado

Durante ciclos anteriores do Living Lab (PRJ002), foram identificados os seguintes problemas recorrentes:

- **Rollbacks manuais complexos**: Procedimentos de reversão dependiam de sequências de comandos CLI sujeitas a erro humano e inconsistências de estado.
- **Falta de rastreabilidade**: Mudanças executadas via comandos avulsos não geravam documentação técnica reutilizável.
- **Tempo de implementação elevado**: Configurações manuais consumiam tempo significativo sem gerar artefatos de valor.
- **Impossibilidade de replicação**: Ambientes não podiam ser recriados com confiança devido à ausência de scripts versionados.
- **Risco de contaminação de estado**: Estados intermediários corrompidos dificultavam diagnóstico e correção.
- **Exposição de credenciais**: Comandos ad-hoc frequentemente continham senhas em histórico de shell ou logs.

### Oportunidade de Melhoria

O PRJ003, como ambiente greenfield em Hyper-V, possui capacidades inexploradas:

1. **Pontos de Verificação Hyper-V**: Permitem rollback instantâneo para estado anterior validado.
2. **Scripts PowerShell/Bash**: Permitem automação completa de configurações com rastreabilidade Git.
3. **Volume Shadow Copy Service (VSS)**: Garante consistência de dados em bancos relacionais durante snapshots.

## Decisão

### Decisão 1: Reversibilidade Obrigatória via Checkpoint Hyper-V

**Princípio:**  
Toda GMUD técnica executada no PRJ003 deve criar um **Ponto de Verificação (Checkpoint)** no Hyper-V antes de iniciar a execução. Este checkpoint torna-se o **Gate de Reversibilidade Oficial**.

**Regras de Aplicação:**

1. **Criação obrigatória**: Antes de qualquer execução técnica, um checkpoint deve ser criado manualmente no Gerenciador Hyper-V.
2. **Tipo de checkpoint**: Deve ser configurado como **"Ponto de Verificação de Produção"** (Production Checkpoint) em vez de "Ponto de Verificação Padrão".
3. **Nomenclatura padronizada**: `PRE-GMUD-XXX` (onde XXX é o número da GMUD).
4. **Validação explícita**: Scripts de execução devem incluir pausa manual para confirmação de criação do checkpoint.
5. **Critério de rollback**: Qualquer falha técnica, validação negativa ou não conclusão de critério de sucesso aciona aplicação do checkpoint.
6. **Captura de evidências pré-rollback**: Antes de aplicar o checkpoint, executar coleta de logs de erro para análise de causa raiz.
7. **Documentação**: Plano de rollback em GMUDs deve referenciar o checkpoint como procedimento oficial.

**Justificativa Técnica — Pontos de Verificação de Produção:**

Os **Production Checkpoints** utilizam:
- **VSS (Volume Shadow Copy Service)** no Windows para congelamento de sistema de arquivos.
- **Backup Writer do PostgreSQL** que garante flush de transações pendentes antes do snapshot.
- **Estado consistente do banco de dados**, simulando um backup a quente (hot backup) corporativo.

Isso evita corrupção de tabelas do PostgreSQL em caso de rollback, crítico para integridade de dados de identidade no midPoint.

**Exemplo de Validação em Script:**

```powershell
Write-Host "ATENÇÃO: Garanta que o Checkpoint 'PRE-GMUD-XXX' foi criado no Hyper-V!" -ForegroundColor Red -BackgroundColor White
Write-Host "TIPO OBRIGATÓRIO: Ponto de Verificação de Produção (Production Checkpoint)" -ForegroundColor Yellow
Read-Host "Pressione ENTER para continuar se o Checkpoint estiver pronto..."
```

**Procedimento de Captura de Evidências Pré-Rollback:**

```powershell
# Antes de aplicar o checkpoint, capturar logs de erro
Write-Host "Capturando evidências de falha antes do rollback..." -ForegroundColor Yellow
ssh $VM_USER@$VM_IP "docker logs midpoint --tail 100 > /srv/prj003/evidencias/erro-gmud-xxx.log 2>&1"
ssh $VM_USER@$VM_IP "journalctl -u docker -n 50 > /srv/prj003/evidencias/erro-sistema-gmud-xxx.log"

Write-Host "Evidências capturadas. Prossiga com rollback via Hyper-V." -ForegroundColor Red
```

**Justificativa:**

- Elimina procedimentos manuais de limpeza sujeitos a erro
- Garante estado idempotente e limpo para novas tentativas
- Reduz tempo de rollback de minutos para segundos
- Permite múltiplas execuções sem contaminação residual
- Facilita diagnóstico preservando logs do estado falho
- **Garante integridade de dados do PostgreSQL** durante rollback

### Decisão 2: Priorização de Infrastructure as Code (IaC)

**Princípio:**  
Toda implementação técnica no PRJ003 deve priorizar a criação e execução de **scripts automatizados** (PowerShell, Bash, Docker Compose, Ansible) em detrimento de comandos CLI avulsos ou configurações manuais via interface gráfica.

**Regras de Aplicação:**

1. **Scripts como artefato primário**: Comandos executados devem estar consolidados em arquivos `.ps1`, `.sh`, `.yml` versionados.
2. **Execução via script**: GMUDs técnicas devem executar scripts, não listas de comandos ad-hoc.
3. **Documentação inline**: Scripts devem conter comentários suficientes para compreensão sem documentação externa.
4. **Versionamento obrigatório**: Scripts devem ser armazenados em repositório Git ou Obsidian com controle de versão.
5. **Reutilização como critério**: Scripts devem ser projetados para execução em contextos diferentes (outras VMs, ambientes).
6. **Evidências automáticas**: Scripts devem gerar logs e evidências técnicas sem intervenção manual.
7. **Idempotência obrigatória**: Scripts devem poder ser executados múltiplas vezes sem causar inconsistências ou falhas.
8. **Gestão segura de credenciais**: **Credenciais hardcoded são estritamente proibidas** em scripts, arquivos YAML ou qualquer artefato versionado.

**Hierarquia de Preferência:**

| Prioridade | Método | Uso Recomendado | Restrições de Segurança |
|-----------|--------|-----------------|------------------------|
| 1ª | Docker Compose / Kubernetes YAML | Deploy de aplicações e serviços | Credenciais via `.env` protegido, nunca em `docker-compose.yml` |
| 2ª | Scripts PowerShell / Bash | Automação de configuração de sistema | Variáveis sensíveis via parâmetros de execução |
| 3ª | Ansible / Terraform | Provisionamento de infraestrutura complexa | Uso de Ansible Vault ou variáveis de ambiente |
| 4ª | Comandos CLI documentados | Apenas para diagnóstico ou investigação | Não permitido para execução de GMUD |
| 5ª | Interfaces gráficas | Não recomendado | Apenas para operações únicas de administração |

**Regras de Gestão de Segredos:**

**Proibido:**
```yaml
# NUNCA faça isso
environment:
  POSTGRES_PASSWORD: <SENHA_EXEMPLO> # ❌ Credencial exposta
```

**Recomendado:**
```yaml
# Opção 1: Arquivo .env (não versionado)
environment:
  POSTGRES_PASSWORD: ${POSTGRES_PASSWORD}  # ✅ Injeta de .env

# Opção 2: Variável de ambiente do host
environment:
  POSTGRES_PASSWORD: ${MIDPOINT_DB_PASS}  # ✅ Definida externamente
```

**Arquivo `.env` (exemplo não versionado):**
```bash
POSTGRES_PASSWORD=<senha_gerada_localmente>
MIDPOINT_REPOSITORY_DATABASE_PASSWORD=<senha_gerada_localmente>
```

**`.gitignore` obrigatório:**
```
.env
*.env
secrets/
```

**Justificativa:**

- **Redução de tempo**: Execução automatizada é mais rápida que configuração manual.
- **Documentação automática**: Script é documentação executável e versionável.
- **Reprodutibilidade garantida**: Ambientes podem ser recriados com fidelidade.
- **Redução de erros**: Eliminação de variação humana na execução.
- **Reuso entre GMUDs**: Scripts podem ser adaptados para mudanças futuras.
- **Facilita testes**: Scripts podem ser executados em ambientes de teste antes de produção.
- **Segurança IAM**: Credenciais não são expostas em versionamento ou logs.
- **Idempotência**: Múltiplas execuções não causam efeitos colaterais indesejados.

## Análise de Causa Raiz Obrigatória (ISO/IEC 27001:2022 A.8.32)

O controle **A.8.32 (Gestão de Mudanças)** da ISO/IEC 27001:2022 exige não apenas a capacidade de rollback, mas o **entendimento da causa raiz** da falha para evitar reincidência.

**Fluxo obrigatório pós-falha:**

1. **Durante a falha**: Script detecta erro crítico e interrompe execução.
2. **Pré-rollback**: Executar captura de logs de erro (Docker, journalctl, aplicação).
3. **Aplicação do checkpoint**: Reverter VM ao estado PRE-GMUD-XXX via Hyper-V.
4. **Análise de causa raiz**: Executor analisa logs capturados e identifica:
   - Erro de configuração
   - Dependência não satisfeita
   - Falha de rede/conectividade
   - Bug de software
   - Problema de permissões
5. **Documentação obrigatória**: Atualizar REL-GMUD com:
   - Descrição da falha
   - Causa raiz identificada
   - Ação corretiva aplicada
   - Validação de que a correção resolve o problema

**Exemplo de Análise de Causa Raiz em REL-GMUD:**

```markdown
## Incidentes e Análise de Causa Raiz

| Item | Descrição |
|------|-----------|
| **Falha Detectada** | Container midPoint não iniciou (exit code 1) |
| **Logs Capturados** | `/srv/prj003/evidencias/erro-gmud-005.log` |
| **Causa Raiz** | Variável de ambiente `MIDPOINT_REPOSITORY_DATABASE_URL` com sintaxe incorreta |
| **Ação Corretiva** | Corrigir JDBC URL de `postgres:5432` para `postgresql://postgres:5432/midpoint` |
| **Validação** | Segunda execução bem-sucedida após correção |
| **Lição Aprendida** | Validar sintaxe JDBC antes do deploy; incluir teste de conectividade no script |
```

### Impactos Positivos

| Área | Benefício |
|------|-----------|
| **Governança** | Rollbacks auditáveis e previsíveis |
| **Operação** | Redução drástica de tempo de rollback |
| **Documentação** | Scripts servem como documentação técnica viva |
| **Aprendizado** | Scripts reutilizáveis em projetos futuros |
| **Qualidade** | Redução de erros de execução manual |
| **Velocidade** | Implementações mais rápidas e confiáveis |
| **Replicação** | Capacidade de recriar ambientes em outros contextos |
| **Segurança IAM** | Credenciais não expostas em versionamento |
| **Integridade de Dados** | Production Checkpoints garantem consistência do PostgreSQL |
| **Compliance** | Evidências prontas para auditoria ISO 27001 |
| **Melhoria Contínua** | Análise de causa raiz previne reincidência de falhas |

### Impactos Negativos e Mitigações

| Impacto Negativo | Mitigação |
|------------------|-----------|
| Tempo inicial maior para criar scripts | Investimento se paga em reuso e confiabilidade |
| Checkpoints consomem espaço em disco | Gerenciar apenas 1-2 checkpoints ativos; Production Checkpoints são menores |
| Necessidade de conhecimento em scripting | Templates de scripts disponíveis para GMUDs futuras |
| Rollback via checkpoint perde logs pós-falha | Coleta obrigatória de evidências antes do rollback |
| Configuração de Production Checkpoints mais complexa | Documentado em procedimento padrão de criação de checkpoint |
| Gestão de `.env` exige processo adicional | Template de `.env.example` versionado; `.env` real em `.gitignore` |

## Aplicação em GMUDs

### Checklist de Aderência ao ADR-001 (v1.1)

Toda GMUD técnica deve validar:

**Reversibilidade:**
- [ ] Checkpoint criado antes da execução?
- [ ] Checkpoint nomeado conforme padrão `PRE-GMUD-XXX`?
- [ ] Checkpoint configurado como **"Ponto de Verificação de Produção"**?
- [ ] Procedimento de captura de logs pré-rollback documentado?

**Infrastructure as Code:**
- [ ] Script de execução criado (não apenas lista de comandos)?
- [ ] Script gera evidências técnicas automaticamente?
- [ ] Script é reutilizável em outros contextos?
- [ ] Script está versionado e armazenado no projeto?
- [ ] Script validado em termos de **idempotência** (pode rodar 2x sem quebrar)?

**Segurança:**
- [ ] Script livre de **credenciais/senhas expostas** no código (hardcoded)?
- [ ] Credenciais sensíveis em arquivo `.env` protegido ou variáveis de ambiente?
- [ ] Arquivo `.env` incluído no `.gitignore`?

**Conformidade:**
- [ ] Plano de rollback referencia o checkpoint?
- [ ] Análise de causa raiz prevista em caso de falha?
- [ ] REL-GMUD preparado para documentar incidentes?

## Exemplos de Aplicação

### Exemplo 1: GMUD-005 (Deploy midPoint) — Revisado

**Checkpoint:**  
- Nome: `PRE-GMUD-005`
- Tipo: **Ponto de Verificação de Produção**
- Criado antes do deploy do Docker Compose

**Script:**  
- Arquivo: `deploy-gmud-005.ps1`
- Funcionalidades:
  - Validação de checkpoint (com alerta de Production Checkpoint)
  - Criação de estrutura de diretórios
  - Deploy do `docker-compose.yml` (credenciais via `.env`)
  - Validação HTTP automática
  - Coleta de evidências técnicas
  - Captura de logs em caso de falha

**Gestão de Segredos:**
```yaml
# docker-compose.yml
services:
  postgres:
    environment:
      POSTGRES_PASSWORD: ${POSTGRES_PASSWORD}  # Injetado de .env

  midpoint:
    environment:
      MIDPOINT_REPOSITORY_DATABASE_PASSWORD: ${MIDPOINT_DB_PASS}
```

```bash
# .env (não versionado)
POSTGRES_PASSWORD=<gerado_localmente>
MIDPOINT_DB_PASS=<gerado_localmente>
```

**Rollback:**  
1. Capturar logs: `docker logs midpoint > evidencias/erro-gmud-005.log`
2. Aplicar checkpoint `PRE-GMUD-005` no Hyper-V
3. Analisar logs e documentar causa raiz em REL-GMUD-005
4. Corrigir script e reexecutar

### Exemplo 2: GMUD Futura (Integração com AD)

**Checkpoint:**  
- Nome: `PRE-GMUD-XXX`
- Tipo: **Ponto de Verificação de Produção**
- Criado antes da configuração do conector AD

**Script esperado:**  
- Arquivo: `deploy-ad-connector.sh`
- Funcionalidades:
  - Upload de configuração XML para midPoint (credenciais via variáveis de ambiente)
  - Teste de conectividade LDAP
  - Validação de sincronização inicial
  - Geração de relatório de correlação
  - Idempotência: detecta se conector já existe antes de criar

**Gestão de Segredos:**
```bash
# Script recebe credenciais via parâmetros
./deploy-ad-connector.sh --ad-user "$AD_SERVICE_USER" --ad-pass "$AD_SERVICE_PASS"

# Credenciais não aparecem em logs
set +x  # Desabilita echo de comandos durante autenticação
```

**Rollback:**  
1. Capturar configuração XML antes de aplicar checkpoint
2. Aplicar checkpoint `PRE-GMUD-XXX`
3. Análise de causa raiz (ex.: porta LDAP bloqueada)
4. Configuração de conector é revertida automaticamente

## Decisões Formais Congeladas

Este ADR congela as seguintes decisões arquiteturais:

1. **Checkpoint Hyper-V é o mecanismo oficial de rollback** para GMUDs técnicas no PRJ003.
2. **Production Checkpoints são obrigatórios** para garantir integridade de dados do PostgreSQL.
3. **Scripts IaC são obrigatórios** para qualquer mudança técnica que envolva configuração de sistema.
4. **Comandos CLI avulsos não são aceitos** como procedimento de execução de GMUD.
5. **GMUDs sem checkpoint pré-criado não devem ser executadas**.
6. **GMUDs sem script de automação não serão aprovadas**.
7. **Credenciais hardcoded são estritamente proibidas** em qualquer artefato versionado.
8. **Análise de causa raiz é obrigatória** para toda falha de GMUD antes de nova tentativa.
9. **Idempotência de scripts é requisito de qualidade** para aprovação de GMUD.
10. **Captura de logs pré-rollback é procedimento obrigatório** para preservar evidências.
