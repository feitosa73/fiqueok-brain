### 🔴 Diagnóstico Técnico: WSL2 não é plataforma enterprise

O ADR-006 documenta perfeitamente as **limitações estruturais do WSL2** para workloads críticos: [ppl-ai-file-upload.s3.amazonaws](https://ppl-ai-file-upload.s3.amazonaws.com/web/direct-files/attachments/69453806/b2559dad-98bc-421a-9c55-285f9f362c38/ADR-006-Revisao-da-Decisao-Arquitetural-Falha-da-Plataforma-WSL2-para-HashiCorp-Vault.md)

- **Systemd parcial**: Serviços de rede não persistem entre reboots
- **Stack de rede virtualizada**: Conflitos persistentes com Tailscale (erro `address already in use`)
- **Raft Storage instável**: Log mostra `Raft RPC layer closed` devido a oscilações de rede [ppl-ai-file-upload.s3.amazonaws](https://ppl-ai-file-upload.s3.amazonaws.com/web/direct-files/attachments/69453806/5691f15a-41e6-4303-b29f-c6b8564148f1/Historico-de-Problemas-com-a-solucao-Hashicorp-em-WSL.txt)
- **Daemon lifecycle problemático**: Processos morrem aleatoriamente ao fechar sessão WSL

**Veredito técnico**: Nenhuma IA conseguiu prever isso porque o WSL2 **nunca foi projetado para serviços de infraestrutura 24/7**. Ele é um ambiente de desenvolvimento, não produção. [developer.hashicorp](https://developer.hashicorp.com/vault/docs/concepts/production-hardening)

### **Proposta Copilot: Hyper-V Gen1 (Temporária)**

**Avaliação técnica**:
- ✅ **Estabilidade comprovada**: Evidências anedóticas mostram Gen1 mais estável que Gen2 para Linux em ambientes com UEFI corrompido [virtualizationdojo](https://virtualizationdojo.com/hyper-v/comparing-hyper-v-generation-1-2-virtual-machines/)
- ✅ **Zero custo, zero dependências externas**
- ✅ **Persistência real**: Systemd, kernel, rede TUN/TAP nativos
- ❌ **Sem Auto-Unseal nativo**: Exige cloud KMS ou workaround com scripts (risco de segurança)
- ⚠️ **Legado**: Gen1 não tem Secure Boot/TPM [nakivo](https://www.nakivo.com/blog/hyper-v-generation-1-vs-2/)

**Parecer de risco**: Solução tática sólida, mas exige migração futura planejada.

### **Caminho Híbrido: Gen1 + OCI Auto-Unseal**

**Por quê?**

1. **Hyper-V Gen1 resolve o problema imediato**:
   - Elimina 100% das falhas do WSL2 (Tailscale, Raft, systemd)
   - Você mantém controle local, sem latência de rede
   - Funciona **hoje**, sem depender de reinstalação do Windows

2. **OCI KMS só para Auto-Unseal** (não para hospedar o Vault):
   - Você configura o Vault local para usar Transit Auto-Unseal via API OCI
   - A chave fica na nuvem (gratuita), mas o Vault roda localmente [docs.oracle](https://docs.oracle.com/iaas/Content/FreeTier/freetier_topic-Always_Free_Resources.htm)
   - **Zero custo, máxima resiliência**

3. **Alinhamento com GRC**:
   - Gen1 é **tecnicamente adequado** para lab/staging segundo CIS Controls
   - Migração para Gen2 documentada no ADR-007 para Q2/2026 (quando reinstalar Windows) [ppl-ai-file-upload.s3.amazonaws](https://ppl-ai-file-upload.s3.amazonaws.com/web/direct-files/attachments/69453806/b2559dad-98bc-421a-9c55-285f9f362c38/ADR-006-Revisao-da-Decisao-Arquitetural-Falha-da-Plataforma-WSL2-para-HashiCorp-Vault.md)
