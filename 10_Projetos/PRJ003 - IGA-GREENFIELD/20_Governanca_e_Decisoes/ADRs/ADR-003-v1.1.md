## Sumário Executivo

Este ADR estabelece a estratégia de **multiconformidade** do PRJ003, adotando o **cross-mapping** (mapeamento cruzado) de controles técnicos e arquiteturais para múltiplos frameworks de GRC:

- **ISO/IEC 27001:2022** (Sistema de Gestão de Segurança da Informação)
- **CIS Controls v8** (Higiene Cibernética Ativa)
- **NIST Cybersecurity Framework (CSF) 2.0** (Funções de Cibersegurança Organizacional)
- **NIST Risk Management Framework (RMF)** (Gestão de Risco Estratégico para Sistemas)
- **SOC 2 (Trust Services Criteria)** (Confiança e Operação)

A decisão transforma o Living Lab Fiqueok 2.0 de um **laboratório de ferramentas** para uma **plataforma de Engenharia de GRC** de classe mundial, refletindo a realidade de grandes empresas onde conformidade raramente é baseada em um único framework.

---

## Contexto

### Problema Observado

Durante ciclos anteriores do Living Lab (PRJ002), a governança era orientada apenas por boas práticas técnicas sem alinhamento formal a frameworks corporativos. Isso gerou:

- **Falta de linguagem comum** com mercado corporativo (auditores, CISOs, GRC managers).
- **Baixa reusabilidade** de evidências para diferentes auditorias.
- **Dificuldade de demonstração de autoridade** técnica em portfólio LinkedIn/GitHub.
- **Perda de oportunidades de aprendizado aplicado** em gestão de risco e compliance.

### Oportunidade Estratégica

O PRJ003, como ambiente greenfield com governança formal (ADRs, GMUDs, Canvases), possui capacidade única de implementar **controles técnicos que servem simultaneamente múltiplos frameworks**.

**Exemplo prático:**  
Um único controle técnico (script de sanitização de credenciais) pode ser mapeado para:
- ISO 27001:2022 → A.8.29 (Segurança em Desenvolvimento)
- CIS Control 3 → Data Protection (Proteção de dados sensíveis)
- NIST CSF 2.0 → PR.DS-1 (Proteção de dados em repouso)
- NIST RMF → CA-2 (Controle de Avaliação de Segurança)
- SOC 2 → CC6.1 (Confidencialidade de informações)

## Decisão

### Decisão 1: Adoção de Cross-Mapping Obrigatório

**Princípio:**  
Todo controle técnico ou decisão arquitetural documentada no PRJ003 deve ser **mapeada simultaneamente** para os cinco frameworks, quando aplicável.

**Regras de Aplicação:**

1. **ADRs devem conter seção "Mapeamento de Conformidade"** com tabela de controles.
2. **GMUDs técnicas devem referenciar controles** aplicáveis no plano de execução.
3. **REL-GMUDs devem evidenciar compliance** com métricas de eficácia de controles.
4. **Scripts IaC devem incluir comentários** indicando controle atendido (ex.: `# CIS Control 4.1, NIST CSF PR.IP-1`).
5. **Mapeamento é obrigatório apenas para controles relevantes** (não todos os 93 controles da ISO 27001).

### Decisão 2: Frameworks Adotados e Foco de Aplicação

| Framework | Foco Principal | Aplicação no PRJ003 |
|-----------|----------------|---------------------|
| **ISO/IEC 27001:2022** | Sistema de Gestão de Segurança da Informação (ISMS) | Base normativa para decisões arquiteturais e processos de mudança |
| **CIS Controls v8** | Higiene Cibernética Ativa (ações técnicas imediatas) | Configuração segura de Docker, auditoria de segredos, hardening de VM |
| **NIST CSF 2.0** | Funções de Cibersegurança (Govern, Identify, Protect, Detect, Respond, Recover) | Governança de decisões, proteção de dados, resposta a incidentes |
| **NIST RMF** | Gestão de Risco Estratégico (6 passos: Categorize → Monitor) | Ciclo de vida de decisões (ADRs), análise de impacto de GMUDs |
| **SOC 2 (TSC)** | Confiança e Operação (Segurança, Disponibilidade, Confidencialidade) | Evidências de eficácia operacional via REL-GMUDs |

**Justificativa de Seleção:**

- **ISO 27001**: Linguagem universal de auditoria corporativa; requisito em fintechs e grandes empresas.
- **CIS Controls**: Pragmático e técnico; foco em ações de alto impacto (quick wins).
- **NIST CSF 2.0**: Framework voluntário mais usado no setor privado; versão 2.0 (fevereiro 2024) inclui função "Govern" alinhada ao DEC-ID-001.
- **NIST RMF**: Framework obrigatório para governo US; usado por multinacionais com contratos federais.
- **SOC 2**: Padrão para SaaS e serviços de TI; relevante para provedores de identidade.

**Diferença entre NIST CSF e NIST RMF:**

- **NIST CSF 2.0**: Framework **voluntário** com 6 funções organizacionais de alto nível; aplicável a qualquer setor.
- **NIST RMF (SP 800-37)**: Framework **obrigatório** para agências federais US; processo estruturado de 6 etapas para autorização de sistemas.
- **Uso no PRJ003**: CSF para funções e controles operacionais; RMF para processo de gestão de risco de decisões arquiteturais.

## Cross-Mapping de Controles Técnicos (Exemplos)

### Exemplo 1: Gestão de Segredos (ADR-002)

**Controle Técnico:**  
Proibição de credenciais hardcoded em scripts e arquivos YAML; uso obrigatório de `.env` protegido.

**Mapeamento:**

| Framework | Controle | Descrição | Evidência no PRJ003 |
|-----------|----------|-----------|---------------------|
| **ISO 27001:2022** | A.8.29 | Segurança em desenvolvimento e processos de suporte | Scripts validam ausência de credenciais antes de commit Git |
| **CIS Control 3** | Data Protection | Proteção de dados sensíveis contra vazamento | `.gitignore` inclui `.env`; template `.env.example` versionado |
| **NIST CSF 2.0** | PR.DS-1 | Proteção de dados em repouso | Credenciais armazenadas em `.env` local, não em repositório |
| **NIST RMF** | CA-2 | Avaliação de Controles de Segurança | Script de auditoria valida conformidade antes de deploy |
| **SOC 2** | CC6.1 | Confidencialidade de informações sensíveis | REL-GMUD evidencia execução sem exposição de credenciais |

**Evidências Geradas:**
- Script de validação: `scripts/audit-secrets.sh`
- Arquivo `.gitignore` versionado
- Template `.env.example` documentado
- REL-GMUD com checklist de conformidade

### Exemplo 2: Production Checkpoints (ADR-002)

**Controle Técnico:**  
Pontos de Verificação de Produção (VSS + PostgreSQL Writer) para garantir integridade de dados durante rollback.

**Mapeamento:**

| Framework | Controle | Descrição | Evidência no PRJ003 |
|-----------|----------|-----------|---------------------|
| **ISO 27001:2022** | A.8.13 | Backup de informações | Production Checkpoints simulam backup a quente corporativo |
| **CIS Control 11** | Data Recovery | Capacidade de recuperação de dados e sistemas | Rollback testado e documentado em GMUDs |
| **NIST CSF 2.0** | RC.RP-1 | Plano de recuperação executado durante ou após evento | Checkpoints aplicados em caso de falha de GMUD |
| **NIST RMF** | CP-9 | Backup de Sistema de Informação | Checkpoints com nomenclatura rastreável (`PRE-GMUD-XXX`) |
| **SOC 2** | A1.2 | Disponibilidade - Backup e Recuperação | Tempo de rollback < 2 minutos (evidência em REL-GMUD) |

**Evidências Geradas:**
- Checkpoints Hyper-V com timestamp
- Logs de rollback bem-sucedido
- REL-GMUD com métrica de RTO (Recovery Time Objective)

## Implementação Prática no PRJ003

### Template de Mapeamento para ADRs

Todo ADR deve incluir seção de mapeamento:

```markdown
## Mapeamento de Conformidade

| Framework | Controle | Implementação neste ADR |
|-----------|----------|-------------------------|
| ISO 27001:2022 | A.X.X | Descrição |
| CIS Control X | Nome do Controle | Descrição |
| NIST CSF 2.0 | XX.XX-X | Descrição |
| NIST RMF | XX-X | Descrição |
| SOC 2 | CCX.X | Descrição |
```

### Comentários em Scripts IaC

Scripts devem indicar controles atendidos:

```bash
#!/bin/bash
# Script: deploy-gmud-005.sh
# Conformidade: CIS Control 4.1 (Secure Configuration)
# Conformidade: NIST CSF 2.0 PR.IP-1 (Configuration Baseline)
# Conformidade: ISO 27001:2022 A.8.32 (Change Management)

# Validação de ausência de credenciais hardcoded
# Conformidade: CIS Control 3.3 (Data Protection)
# Conformidade: NIST CSF 2.0 PR.DS-1 (Data at Rest)
grep -r "password=" docker-compose.yml && exit 1
```

### Checklist de Conformidade em GMUDs

GMUDs técnicas devem incluir validação:

```markdown
### Checklist de Conformidade

- [ ] Controles ISO 27001 aplicáveis identificados?
- [ ] Controles CIS relevantes implementados?
- [ ] Funções NIST CSF 2.0 atendidas?
- [ ] Etapa NIST RMF aplicável ao risco da mudança?
- [ ] Evidências de eficácia documentadas?
```

## Justificativa Estratégica

### Valor para Aprendizado Aplicado

- **Linguagem corporativa**: Aprende a "falar a língua" de auditores e CISOs.
- **Visão holística de GRC**: Entende como frameworks se complementam (ISO = estratégico; CIS = tático; CSF = operacional).
- **Portfolio diferenciado**: GitHub com evidências de multiconformidade demonstra maturidade técnica.
- **NIST CSF 2.0 atualizado**: Demonstra conhecimento de frameworks lançados em 2024.

### Valor para Autoridade no LinkedIn

- **Posts técnicos fundamentados**: "Como implementei NIST CSF 2.0 PR.DS-1 com Docker Compose no meu Lab".
- **Credibilidade em GRC**: Não apenas IAM técnico, mas engenheiro de compliance.
- **Diferenciação competitiva**: Poucos profissionais combinam IAM + GRC multiframework em portfólio público.

### Realidade de Grandes Empresas

Fintechs e multinacionais precisam atender simultaneamente:
- ISO 27001 (certificação corporativa)
- SOC 2 (exigência de clientes SaaS)
- CIS Controls (benchmarks de segurança)
- NIST CSF 2.0 (framework voluntário mais adotado no setor privado)
- NIST RMF (compliance regulatório em setores específicos)

O PRJ003 simula essa realidade em escala de laboratório.

## Consequências

### Impactos Positivos

| Área | Benefício |
|------|-----------|
| **Aprendizado** | Exposição a linguagem e práticas de GRC corporativo |
| **Portfolio** | GitHub com evidências de multiconformidade (diferenciação competitiva) |
| **Eficiência** | Um controle técnico gera evidências para 5 frameworks simultaneamente |
| **Reusabilidade** | Evidências servem múltiplas auditorias sem retrabalho |
| **Autoridade Técnica** | Demonstra maturidade além de ferramentas (engenheiro de compliance) |
| **Realismo** | Simula ambiente de fintech ou multinacional |

### Impactos Negativos e Mitigações

| Impacto Negativo | Mitigação |
|------------------|-----------|
| Tempo adicional para mapear controles | Template de mapeamento pronto em ADRs |
| Risco de mapeamento incorreto | Revisão por framework oficial (links de referência) |
| Complexidade percebida | Mapeamento opcional para controles menores; obrigatório apenas para ADRs |
| Necessidade de conhecimento dos frameworks | Aprendizado incremental; foco inicial em controles técnicos |
| Possível confusão entre CSF e RMF | Documentação clara de diferenças e aplicações |

## Critérios de Sucesso

Este ADR será considerado bem-sucedido quando:

- 100% dos ADRs incluírem seção de Mapeamento de Conformidade (5 frameworks).
- Scripts IaC conterem comentários de controles aplicáveis (ISO, CIS, NIST CSF).
- REL-GMUDs evidenciarem eficácia de controles (ex.: tempo de rollback, ausência de credenciais expostas).
- Portfolio GitHub demonstrar multiconformidade de forma clara.
- Pelo menos 1 post técnico no LinkedIn explicar cross-mapping de um controle.
- NIST CSF 2.0 ser referenciado em pelo menos 3 GMUDs técnicas.

## Critérios de Revisão

Este ADR poderá ser revisado caso:

1. Novos frameworks relevantes surjam (ex.: DORA para instituições financeiras EU).
2. Frameworks adotados publiquem novas versões (ex.: ISO 27001:2025, CIS v9, NIST CSF 3.0).
3. Mapeamento se mostrar contraproducente (excesso de burocracia sem valor).
4. Evidências de auditoria revelarem gaps de conformidade.
5. NIST CSF 2.0 for substituído por versão mais recente.

Qualquer revisão exige novo ADR formal.

---

## Decisões Formais Congeladas

Este ADR congela as seguintes decisões arquiteturais:

1. **Cross-mapping é obrigatório para ADRs** (ISO 27001, CIS, NIST CSF 2.0, NIST RMF, SOC 2).
2. **NIST CSF 2.0 é adotado como framework de funções operacionais** complementar ao RMF.
3. **Scripts IaC devem incluir comentários de controles** quando aplicável.
4. **REL-GMUDs devem evidenciar eficácia de controles** com métricas.
5. **Mapeamento foca em controles técnicos relevantes**, não todos os controles.
6. **Templates de mapeamento devem ser usados** para uniformidade.
7. **Evidências devem ser reutilizáveis** para múltiplas auditorias.
8. **Linguagem de GRC corporativo é adotada** em documentação técnica.
9. **NIST CSF 2.0 e RMF são complementares**, não substitutos.
